# 第3章 系统设计

## 3.1 系统建设目标

本系统面向轨道交通检修场景，围绕“检修工具数字化管理、人员权限可控、关键操作可追溯”三项核心需求展开设计。系统目标如下：

1. 建立统一的工具管理台账，实现工具信息、库存信息与维护组信息的集中管理；
2. 建立基于角色的权限管理机制，支持管理员维护账号并对员工分配功能权限；
3. 建立系统策略配置能力，包括密码策略、告警阈值、工具类型字典等可配置项；
4. 建立审计能力，对关键业务操作进行日志化记录，支持按条件检索与追踪。

## 3.2 总体架构设计

系统采用前后端分离架构，并通过 Docker Compose 完成环境编排。整体可划分为三层：

1. 表现层：基于 Vue3 + Element Plus 构建后台管理界面；
2. 业务层：基于 FastAPI 构建 REST API，承担业务规则校验、权限控制与数据聚合；
3. 数据层：基于 PostgreSQL 持久化核心业务数据，Redis 与 EMQX 作为扩展能力预留。

该架构具备良好的可扩展性：前端模块可按路由继续拓展，后端 API 可按领域拆分，数据层可通过表结构与索引优化承载更多业务规模。

## 3.3 功能模块设计

### 3.3.1 登录与会话模块

系统提供登录入口，用户通过账号密码进入系统。登录后前端持久化会话信息，并通过路由守卫控制页面访问。当前实现包含管理员账号与员工账号两类用户。

### 3.3.2 工具管理模块

工具管理模块实现了工具数据的全生命周期管理，包含：

- 条件查询（工具编码、工具类型、工具名称、库存）；
- 分页浏览；
- 新增与修改；
- 单条删除与批量删除；
- 导出 CSV。

工具类型字段不再硬编码，而是由“工具类型字典”动态提供，保证业务口径一致。

### 3.3.3 权限管理模块

权限管理模块采用角色分流设计：

1. 管理员视图：可新增账号、维护员工信息（工号、姓名、部门、职位、角色）并分配权限；
2. 员工视图：仅可查看个人信息和已分配权限。

系统支持“管理员不可删除当前登录管理员自身账号”的安全保护逻辑。

### 3.3.4 系统设置模块

系统设置模块包含四个子功能：

1. 密码策略配置；
2. 告警阈值配置；
3. 工具类型字典管理；
4. 操作日志查询。

其中策略配置与字典配置属于系统级参数，默认仅管理员可修改；操作日志用于审计关键操作并支持条件检索。

## 3.4 数据库设计

### 3.4.1 核心实体

系统核心实体包括：

- `tools`：工具实体；
- `users`：账号实体；
- `system_settings`：系统参数实体；
- `tool_types`：工具类型字典实体；
- `operation_logs`：操作日志实体。

### 3.4.2 关键表说明

1. `tools`：保存工具编码、工具类型、名称、库存、维护组与图片字段；
2. `users`：保存账号、工号、姓名、部门、职位、角色、权限数组；
3. `system_settings`：采用 `key-value(JSON)` 形式保存密码策略和阈值策略；
4. `tool_types`：保存类型名称、描述、排序和启用状态；
5. `operation_logs`：记录模块、动作、操作者、目标、明细和时间戳。

### 3.4.3 索引与约束

- 对 `tools.tool_code`、`users.username`、`users.employee_no` 建立唯一约束；
- 对常用检索字段（如模块、操作人、时间）建立索引；
- 对关键业务字段设置非空约束，提升数据完整性。

## 3.5 权限与安全设计

系统基于角色控制核心操作：

1. 管理员可执行用户维护、权限分配、系统参数变更、日志查询；
2. 员工仅可查看自身权限信息。

在实现上，后端通过管理员校验函数统一拦截高权限接口，避免仅依赖前端判断造成的越权风险。

## 3.6 关键业务流程设计

### 3.6.1 管理员新增员工并分配权限流程

1. 管理员进入权限管理页；
2. 提交员工基础信息；
3. 后端校验账号与工号唯一性后入库；
4. 管理员为该员工分配权限；
5. 后端保存权限并写入操作日志。

### 3.6.2 系统设置变更流程

1. 管理员在系统设置页修改策略或字典项；
2. 后端校验管理员身份；
3. 更新配置表；
4. 写入操作日志；
5. 前端刷新读取最新配置。

# 第4章 系统实现

## 4.1 开发环境与运行环境

### 4.1.1 开发环境

- 前端：Vue3、Vite、Element Plus
- 后端：FastAPI、SQLAlchemy、Pydantic
- 数据库：PostgreSQL
- 编排：Docker Compose

### 4.1.2 运行方式

系统支持两种运行方式：

1. 本地方式：分别运行前端与后端，外接数据库；
2. 一键方式：通过 `docker compose up -d --build` 启动全套服务。

## 4.2 前端实现

### 4.2.1 路由守卫与会话实现

前端在登录成功后将会话写入本地存储，路由守卫在页面跳转时检查会话状态，未登录用户统一重定向至登录页。

### 4.2.2 主框架实现

主界面采用“左侧导航 + 顶部信息栏 + 内容区域”的后台布局模式。左侧包含工具管理、权限管理、系统设置三个入口。

### 4.2.3 工具管理页面实现

工具管理页包含检索区、操作按钮区、数据表格区和分页区，并通过弹窗实现新增与编辑。所有数据操作均通过 Axios 调用后端 API 实现。

### 4.2.4 权限管理页面实现

权限管理页依据角色动态渲染：

- 管理员：显示用户表格与新增/编辑/分配权限/删除操作；
- 员工：显示个人信息与权限清单。

### 4.2.5 系统设置页面实现

系统设置页采用 Tab 结构，对应密码策略、告警阈值、工具类型字典、操作日志四个子模块。管理员可编辑并保存配置，日志模块支持分页检索。

## 4.3 后端实现

### 4.3.1 FastAPI 服务组织

后端采用统一路由文件集中管理接口，核心能力分为：

1. 用户与权限接口；
2. 工具管理接口；
3. 系统设置接口；
4. 审计日志接口。

### 4.3.2 ORM 模型实现

使用 SQLAlchemy 定义实体模型并映射到 PostgreSQL 表。启动时自动建表并执行种子数据初始化，降低部署门槛。

### 4.3.3 角色校验实现

高权限接口（如用户管理、系统参数更新、日志查询）统一经过管理员身份校验函数，确保权限控制在服务端生效。

### 4.3.4 审计日志实现

在新增用户、分配权限、修改策略、修改字典、删除账号等关键动作后，系统自动写入操作日志，形成完整审计链路。

## 4.4 数据库实现

### 4.4.1 初始化脚本

数据库初始化脚本中已包含表结构创建、索引创建与默认数据插入逻辑，可在新环境快速恢复基础运行态。

### 4.4.2 数据一致性策略

- 对编码类字段使用唯一约束防止重复；
- 对关键字段使用非空约束防止脏数据；
- 对外键依赖较弱的字典关系通过业务校验保证有效性。

## 4.5 联调与验证

### 4.5.1 接口验证

通过接口联调验证了如下能力：

- 工具管理增删改查与分页检索；
- 权限管理新增用户、分配权限、删除用户；
- 系统设置策略保存与字典管理；
- 操作日志查询。

### 4.5.2 页面验证

前端页面已验证管理员与员工两类账号在权限管理、系统设置中的可见性与操作边界。

## 4.6 本章小结

本章完成了系统从前端页面、后端服务到数据库设计的实现说明，展示了项目在“业务功能、权限控制、配置治理、审计追踪”四个方面的完整落地路径，为后续部署优化与性能扩展提供了基础。
