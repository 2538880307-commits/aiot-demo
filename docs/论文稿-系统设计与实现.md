# 第3章 系统设计（更新版）

## 3.1 系统建设目标

本系统面向轨道交通检修作业，围绕“工具可管、权限可控、过程可追溯、异常可识别”四项目标建设：

1. 对检修工具建立统一数字化台账，支持全生命周期维护。
2. 建立基于角色与细粒度权限的访问控制机制。
3. 建立系统参数治理能力，支持密码策略、阈值、字典配置。
4. 基于图像识别实现检修前后工具数量自动对比，辅助发现缺失风险。

## 3.2 总体架构设计

系统采用前后端分离架构，并通过 Docker Compose 编排运行环境。整体分为三层：

1. 表现层：Vue3 + Element Plus 构建管理端。
2. 业务层：FastAPI 提供 REST API，承担鉴权、校验、日志写入和识别编排。
3. 数据层：PostgreSQL 持久化业务数据，Redis 和 EMQX 提供扩展支撑。

该架构支持后续按业务域继续扩展接口、页面与数据模型。

## 3.3 功能模块设计

### 3.3.1 登录与权限模块

系统通过后端接口 `POST /api/v1/auth/login` 完成账号密码校验。登录成功后，前端持久化会话信息，并基于路由元信息（`meta.permission`）控制模块访问。管理员默认拥有全部权限，普通员工仅可访问已授权模块。

### 3.3.2 工具管理模块

工具管理模块提供工具信息的新增、查询、修改、删除、批量删除与导出。后端在写操作中执行工具编码唯一性校验，并基于工具类型字典校验类型合法性。

### 3.3.3 工具识别模块

工具识别模块支持上传检修前后两张图片，分别调用识别接口 `POST /api/v1/tool-count/detect`，返回总数与分类数量。前端根据两次结果计算差异并输出“缺失/一致/新增”判定。

### 3.3.4 权限管理模块

管理员可维护用户信息并分配功能权限；员工可查看个人信息与权限。系统在后端实现“管理员不可删除当前登录账号”的保护逻辑。

### 3.3.5 系统设置与审计模块

系统设置模块支持密码策略、告警阈值、工具类型字典维护。关键变更统一写入操作日志，管理员可按条件分页查询。

## 3.4 数据库设计

### 3.4.1 核心实体

核心实体包括：`tools`、`users`、`tool_types`、`system_settings`、`operation_logs`。

### 3.4.2 关键字段

1. `tools`：`tool_code`、`tool_type`、`tool_name`、`stock`、`team`。
2. `users`：`username`、`employee_no`、`role`、`permissions`、`password_hash`。
3. `tool_types`：类型名、描述、排序、启用状态。
4. `system_settings`：配置键与 JSON 配置值。
5. `operation_logs`：模块、动作、操作者、目标、明细、时间戳。

### 3.4.3 约束与完整性

- 对工具编码、用户名、工号设置唯一约束。
- 对核心字段设置非空约束。
- 对常用检索场景（如日志时间倒序）进行查询优化。

## 3.5 安全与权限设计

系统采用“后端强校验 + 前端引导控制”双层机制：

1. 后端通过 `require_admin` 和 `require_permission` 统一校验权限。
2. 前端根据权限集合动态展示菜单和路由入口。
3. 高权限操作（用户维护、系统配置）仅管理员可执行。

## 3.6 关键流程设计

### 3.6.1 管理员分配权限流程

1. 管理员创建或编辑员工账号。
2. 管理员提交权限集合。
3. 后端过滤并保存有效权限项。
4. 记录“分配权限”操作日志。

### 3.6.2 工具识别对比流程

1. 上传检修前图片并识别。
2. 上传检修后图片并识别。
3. 计算总数差异与分类差异。
4. 输出风险提示并记录识别日志。

# 第4章 系统实现（更新版）

## 4.1 开发与运行环境

### 4.1.1 开发环境

- 前端：Vue3、Vite、Element Plus
- 后端：FastAPI、SQLAlchemy 2.0、Pydantic
- 数据：PostgreSQL
- 编排：Docker Compose
- 识别：YOLO 模型（`best_tools.pt`）

### 4.1.2 运行方式

1. 容器化方式：`docker compose up -d --build`。
2. 本地方式：分别启动后端与前端，并连接 PostgreSQL。

## 4.2 前端实现

### 4.2.1 路由守卫与权限跳转

路由文件为 `frontend/src/router/index.js`。系统在路由元信息中声明模块权限，未授权用户自动重定向到首个可访问页面或无权限页面。

### 4.2.2 主要页面实现

1. `ToolRecognitionView.vue`：工具管理台账页。
2. `ToolCountView.vue`：检修前后识别对比页。
3. `PermissionManagementView.vue`：用户与权限维护页。
4. `SystemSettingsView.vue`：参数与字典配置页。

## 4.3 后端实现

### 4.3.1 服务启动与初始化

`backend/app/main.py` 在生命周期启动阶段执行自动建表、字段兼容补丁与种子初始化，并启动 MQTT 服务。

### 4.3.2 接口分层

后端核心接口包括：

1. 认证与基础接口（健康、登录、统计）。
2. 工具管理接口。
3. 工具识别接口。
4. 权限管理接口。
5. 系统设置与日志接口。

### 4.3.3 审计日志实现

通过统一 `append_log` 方法记录关键动作，覆盖识别成功/失败、工具变更、用户变更、系统设置变更等事件。

## 4.4 数据库实现

### 4.4.1 建表与种子

启动时通过 ORM 自动建表；当数据库为空时自动写入默认工具、用户与系统配置，降低首次部署复杂度。

### 4.4.2 一致性策略

- 业务主键与编码字段约束避免重复数据。
- 写操作前执行业务规则校验。
- 配置与字典变更同步记录审计日志。

## 4.5 联调验证

### 4.5.1 接口联调

已覆盖工具管理、权限管理、系统设置、识别上传等核心接口链路。

### 4.5.2 页面联调

已验证管理员与员工两类角色的菜单可见性、接口权限边界与识别对比交互流程。

## 4.6 本章小结

当前实现已形成“工具管理 + 识别对比 + 权限治理 + 审计追踪”的闭环基础，为后续在告警联动、识别准确率优化与统一身份认证方面的增强提供了稳定底座。
