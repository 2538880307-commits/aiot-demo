# 轨道交通检修工具监测平台：项目框架与关键实现说明

## 1. 项目目标与总体架构

本项目是一个 AIoT 场景下的 Web 管理平台，围绕“工具监测与权限治理”实现了完整的前后端联动能力。当前版本已实现：

- 登录与角色区分（管理员/员工）
- 工具管理（查询、分页、新增、修改、删除、批量删除、导出）
- 权限管理（管理员维护账号与权限、员工查看自身权限）
- 系统设置（密码策略、告警阈值、工具类型字典、操作日志查询）

整体采用三层架构：

1. 前端展示层：Vue3 + Vite + Element Plus
2. 业务服务层：FastAPI + SQLAlchemy(Async)
3. 数据与中间件层：PostgreSQL + Redis + EMQX（MQTT）

---

## 2. 技术栈与选型说明

### 2.1 前端

- `Vue 3`：组件化开发，适合后台管理系统。
- `Vite`：启动快、构建快，开发体验好。
- `Element Plus`：快速构建表单、表格、弹窗、分页。
- `Axios`：统一 HTTP 请求。

### 2.2 后端

- `FastAPI`：异步 API，自动生成 Swagger 文档。
- `SQLAlchemy 2.0 + asyncpg`：异步 ORM 访问 PostgreSQL。
- `Pydantic`：请求/响应模型校验。
- `paho-mqtt`：连接 MQTT Broker，为后续边缘消息接入预留。

### 2.3 基础设施

- `PostgreSQL`：业务主数据存储。
- `Redis`：缓存/队列扩展能力预留。
- `EMQX`：MQTT Broker，接边缘设备。
- `Docker Compose`：本地一键编排运行。

## 4. 运行链路（核心数据流）

### 4.1 登录与角色分流

1. 前端登录页校验本地默认账号（`admin/operator`）。
2. 登录成功后写入 `localStorage` 会话（含 `username/roleKey`）。
3. 路由守卫根据会话判断是否可进入业务页面。
4. 权限管理页根据 `roleKey` 分流：
   - `admin`：显示用户管理与权限分配
   - `employee`：显示“我的权限”

### 4.2 工具管理业务流

1. 前端筛选条件发起 `GET /api/v1/tools`。
2. 后端基于 SQLAlchemy 动态拼接查询条件并分页。
3. 新增/编辑提交至 `POST/PUT /api/v1/tools...`。
4. 后端校验工具编码唯一性、工具类型有效性后落库。
5. 删除、批量删除成功后刷新表格。

### 4.3 系统设置业务流

1. 前端系统设置四个 Tab 分别调用对应 API。
2. 管理员可以保存密码策略、告警阈值、工具类型字典。
3. 后端将配置落到 `system_settings` 或 `tool_types`。
4. 关键操作同步写入 `operation_logs`。
5. 操作日志页按模块/动作/操作人检索并分页展示。

## 5. 数据库模型设计（核心表）

### 5.1 `tools`（工具表）

- `tool_code`：唯一工具编码
- `tool_type`：工具类型（受字典约束）
- `tool_name`：工具名称
- `stock`：库存
- `team`：维护组
- `image_url`：图片数据

### 5.2 `users`（账号表）

- `username`：登录账号（唯一）
- `employee_no`：工号（唯一）
- `name/department/position`
- `role`：`admin | employee`
- `permissions`：JSON 数组

### 5.3 `system_settings`（系统配置）

- `setting_key`：配置键（如 `password_policy`）
- `setting_value`：JSON 配置内容
- `updated_by/updated_at`

### 5.4 `tool_types`（工具类型字典）

- `name`：类型名（唯一）
- `description`
- `sort_order`
- `enabled`

### 5.5 `operation_logs`（操作日志）

- `module/action/actor/target`
- `detail_json`
- `timestamp`

## 6. 后端关键实现说明

### 6.1 应用启动与自动建表

文件：`backend/app/main.py`

关键点：

- `Base.metadata.create_all`：启动时自动创建模型对应表。
- 执行种子函数：
  - `seed_tools_if_empty`
  - `seed_users_if_empty`
  - `seed_system_configs`

这样在新环境里无需手动建表即可启动。

### 6.2 管理员鉴权逻辑

文件：`backend/app/api/routes.py`

`require_admin(requester_username, db)` 统一做：

- 请求用户存在性校验
- 角色是否为 `admin` 校验

管理员专属接口（用户管理、系统设置写操作、日志查询）都复用该方法。

### 6.3 工具管理 API

- `GET /api/v1/tools`：多条件 + 分页查询
- `POST /api/v1/tools`：新增（编码唯一 + 类型有效）
- `PUT /api/v1/tools/{id}`：修改
- `DELETE /api/v1/tools/{id}`：删除
- `POST /api/v1/tools/batch-delete`：批量删除

### 6.4 权限管理 API

- `GET /api/v1/users`：管理员查询用户
- `POST /api/v1/users`：管理员新增用户
- `PUT /api/v1/users/{id}`：管理员改用户资料
- `PUT /api/v1/users/{id}/permissions`：管理员分配权限
- `DELETE /api/v1/users/{id}`：管理员删除账号（禁止删自己）
- `GET /api/v1/users/me`：员工查看自身资料
- `GET /api/v1/users/me/permissions`：员工查看自身权限

### 6.5 系统设置 API

- `GET/PUT /api/v1/settings/password-policy`
- `GET/PUT /api/v1/settings/alert-threshold`
- `GET /api/v1/settings/tool-types`
- `GET /api/v1/settings/tool-types/options`
- `POST/PUT/DELETE /api/v1/settings/tool-types...`
- `GET /api/v1/settings/operation-logs`

并在关键写操作后调用 `append_log(...)` 记录审计日志。

## 7. 前端关键实现说明

### 7.1 登录与会话

- `frontend/src/auth/defaultAccounts.js`：本地默认账号
- `frontend/src/auth/session.js`：`localStorage` 持久化
- `frontend/src/router/index.js`：登录守卫

### 7.2 主界面框架

文件：`frontend/src/views/MainLayout.vue`

- 左侧菜单：工具管理、权限管理、系统设置
- 顶部展示：页面标题 + 当前登录用户 + 退出

### 7.3 工具管理页

文件：`frontend/src/views/modules/ToolRecognitionView.vue`

关键点：

- 列表、筛选、分页完全走后端 API
- 新增/修改弹窗 + 图片上传前端预览
- 导出 CSV
- 工具类型下拉改为 `GET /api/v1/settings/tool-types/options` 动态获取

### 7.4 权限管理页

文件：`frontend/src/views/modules/PermissionManagementView.vue`

- 管理员视图：用户 CRUD + 权限分配
- 员工视图：只读“我的权限”

### 7.5 系统设置页

文件：`frontend/src/views/modules/SystemSettingsView.vue`

- 四个 Tab 分别对应四类配置能力
- 管理员可保存；员工默认只读
- 操作日志支持筛选分页

## 8. 容器化与环境配置

### 8.1 容器服务

`docker-compose.yml` 包含：

- `postgres`
- `redis`
- `mqtt (emqx)`
- `backend`
- `frontend`

### 8.2 代理与联调

前端通过 Vite 代理转发 API，Compose 中注入：

- `VITE_API_PROXY_TARGET=http://backend:8000`

避免容器内 `localhost` 指向错误导致“保存失败”。

## 10. 已知约束与后续建议

### 10.1 当前约束

- 登录鉴权仍是“前端默认账号校验 + 查询参数传 `requester_username`”，非生产级安全方案。
- 图片存储目前为 base64/url 字段，未做对象存储。
- 操作日志未接入真实 IP/设备信息。

### 10.2 建议迭代

1. 接入 JWT + 后端鉴权中间件（替代 query 参数身份传递）
2. 引入 Alembic 管理迁移（替代 `create_all`）
3. 图片上传改对象存储（MinIO/OSS）
4. 增加密码哈希、登录失败审计、RBAC 细粒度权限
5. 对系统设置增加“变更历史回滚”

---

## 11. 快速验证清单

1. 管理员登录后进入权限管理：可新增/修改/删除用户并分配权限。
2. 员工登录后进入权限管理：仅看到自己的权限。
3. 系统设置里修改工具类型字典后，工具管理页下拉立即反映。
4. 系统设置保存后，操作日志页可查询到对应记录。
